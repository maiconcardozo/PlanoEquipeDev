# Processo de Abertura e Tratamento de Bug pelo QA e em Produção

## Objetivo

Definir o fluxo para abertura, sinalização e tratamento de bugs identificados tanto pelo QA em ambiente de testes quanto por usuários/suporte em ambiente de produção, garantindo rastreabilidade, priorização e comunicação eficaz.

---

## 1. Bug Identificado pelo QA (em ambiente de testes)

### 1.1 Identificação do Bug

- O QA identifica um bug durante os testes de uma User Story (US) específica.

### 1.2 Criação do Bug

- O QA deve criar um novo item do tipo **Bug** na ferramenta de gestão (ex: Azure DevOps, Jira, etc).
- O Bug **deve ser criado como filho da User Story (US)** que está sendo testada.
- No descritivo do Bug, incluir:
  - Passos para reprodução
  - Comportamento esperado vs. comportamento observado
  - Evidências (prints, vídeos, logs, etc)
  - Severidade/impacto

### 1.3 Sinalização ao Desenvolvedor

- Assim que criado o Bug, o QA deve imediatamente:
  - Sinalizar o desenvolvedor responsável pela US (marcação direta ou menção).
  - Garantir que o Bug esteja visível para o time e priorizado adequadamente.

### 1.4 Tag na User Story

- O QA deve **adicionar a tag "AGUARDANDO DEFEITO"** na User Story correspondente.
- Esta tag indica que a entrega está aguardando correção de bug para prosseguir.

### 1.5 Interrupção do Desenvolvimento da Task

- O desenvolvedor deve:
  - **Bloquear a task em andamento** (status Doing/Em Progresso) e sinalizar no board que está aguardando correção do Bug.
  - Priorizar a correção do Bug antes de seguir com o desenvolvimento da task original.

### 1.6 Correção e Reteste

- O desenvolvedor corrige o Bug e atualiza o status para "Pronto para Teste" ou equivalente.
- O QA realiza novo teste para validar a correção.
  - Se o Bug persistir, reabrir ou criar novo Bug conforme a análise.
  - Se resolvido, encerrar o Bug, remover a tag "AGUARDANDO DEFEITO" da US e liberar a task original.

### 1.7 Atualização na Daily

- **Todo o fluxo de bugs abertos, em andamento e resolvidos deve ser atualizado e comunicado na daily.**
  - O time deve revisar quais US estão com a tag "AGUARDANDO DEFEITO".
  - QA e Dev devem alinhar pendências, prioridades e status de correção de bugs diariamente.

## 2. Bug Identificado em Produção (PROD)

### 2.1 Identificação do Bug em Produção

- O bug é identificado por usuários finais, clientes, equipe de suporte ou por monitoramentos automáticos em ambiente de produção.

### 2.2 Criação e Priorização do Bug

- Um item do tipo **Bug** deve ser imediatamente criado na ferramenta de gestão.
- O Bug deve conter:
  - Descrição detalhada do problema.
  - Evidências (prints, logs, relatórios de erro, etc).
  - Impacto no negócio e severidade do problema.
  - Passos para reprodução, se aplicável.
- Relacione o Bug à User Story associada, se existir; caso não exista, relacione ao épico ou área do sistema correspondente.
- O Bug deve ser priorizado como **urgente** no board, sinalizando a necessidade de resposta imediata.
- Recomenda-se utilizar uma tag como **URGENTE** ou **BUG EM PROD** para facilitar a visualização.
- Considere mover a US para uma **swimlane** específica (linha horizontal de separação no Azure DevOps) para Bugs em Produção.

### 2.3 Comunicação e Tratamento

- A existência do Bug em produção deve ser comunicada a todo o time na daily e, se necessário, em outros canais de comunicação de emergência.
- O desenvolvedor responsável deve iniciar a correção imediatamente, bloqueando outras demandas até a resolução do Bug em produção.
- Realizar testes locais e, se possível, homologação antes do deploy em produção.
- Após correção, realizar o deploy em produção seguindo o fluxo de publicação segura do time.
- Validar junto ao QA e/ou solicitante se o problema foi efetivamente resolvido em produção.
- Atualizar o status do Bug e remover tags de urgência.
- Registrar lições aprendidas e, caso necessário, atualizar documentação ou implementar melhorias preventivas.

---

## Resumo do Fluxo

### Bug pelo QA:
1. QA identifica o Bug.
2. QA abre Bug filho da US.
3. QA sinaliza o dev responsável.
4. QA adiciona a tag "AGUARDANDO DEFEITO" na US.
5. Dev bloqueia a task e corrige o Bug.
6. QA retesta, remove a tag da US e encerra Bug/task.
7. Todo o fluxo é revisado e atualizado na daily.

### Bug em Produção:
1. Identificação do Bug em produção (usuário/suporte/monitoramento).
2. Bug é registrado na ferramenta com prioridade urgente.
3. Time é sinalizado e prioriza o Bug (tag "BUG EM PROD" ou "URGENTE").
4. US pode ser movida para swimlane específica de produção no Azure DevOps.
5. Correção imediata, testes e deploy.
6. Comunicação e validação pós-correção.
7. Atualização na daily e registro de lições aprendidas.

---

## Observações

- Todo Bug deve ter vínculo claro com a User Story de origem para rastreabilidade.
- Bugs em produção têm prioridade máxima e exigem comunicação imediata.
- O uso de **swimlanes** no Azure DevOps ajuda a destacar e gerenciar visualmente bugs críticos ou em produção.
- O fluxo visa garantir qualidade, evitar retrabalho, manter previsibilidade nas entregas e promover alinhamento diário do time.
